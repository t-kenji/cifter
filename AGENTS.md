# Cifter エージェント運用規約

対象読者: エージェントAI
この文書が正本として扱う範囲: エージェントの行動規約、意思決定手順、仕様参照順

## 1. 役割

この文書は、エージェント AI が Cifter リポジトリで作業するときの運用規約を定義する。
技術仕様の正本は `docs/specs/overview.md` とその参照先に置く。

## 2. 仕様参照順

実装・判断の前に、次をこの順序で確認する。

1. `docs/specs/overview.md`（エージェント AI 向けの文書体系と更新ルール）
2. `docs/specs/configuration.md`
3. `docs/specs/data-model/`
4. `docs/specs/pipeline/`
5. `docs/overview.md`（開発者向けの文書体系と更新ルール）

## 3. 意思決定ポリシー

- 仕様が曖昧または不足している場合、独断で実装を開始しない。
- 人間へ短く提示する内容は次に限定する。
  1. 選択肢
  2. 利点・欠点および欠点の重大性
  3. 影響範囲
  4. 推奨案と理由
- 仕様解釈を固定したら、先に `docs/specs/` を更新してから実装する。

## 4. 言語ポリシー

自然言語で記述する説明・コメント・回答は日本語を使用する。

## 5. 作業原則

- 仕様にない判断を独断で追加しない。
- 変更範囲は必要十分に限定する。
− 特に指示がない場合、仕様・設計までで作業を区切り、承認後に実装に着手する。
- 価値の低い実現方法（暫定的な実装、一時的な実装、妥協の実装）を自動的に採用しない。
- 利用者向け説明は `README.md`、開発者向け説明は `docs/`、実行可能仕様は `docs/specs/` に置く。
- 利用者向けドキュメントはシンプルかつコンパクトな記述を採用し、できるだけ安易な表現を使う。
- 開発者向けドキュメントは抽象的かつ網羅的に記述し、すべての機能について人間可読性を優先した表現を使う。

## 6. 標準手順

1. `docs/specs/overview.md` 起点で関連仕様を読む。
2. 変更タスクを分解し、差分を最小化する。
3. 実装・テスト・ドキュメントを整合させる。
4. 仕様変更を伴う場合は `docs/specs/` を先に更新し、その後に `docs/` と `README.md` を必要最小限で追従させる。

## 7. 推奨コマンド

- 依存導入: `uv sync`
- テスト: `uv run pytest`
- Lint: `uv run ruff check .`
- Format: `uv run ruff format .`
- Type Check: `uv run ty check .`

## 8. 完了条件

- 仕様にない判断をしていない。
- 変更内容が `docs/specs/` 系と整合している。
- 解析結果と生成結果のトレーサビリティを維持している。

## 9. 仕様との齟齬

仕様と実装が衝突した場合は、実装を黙って曲げず、先に `docs/specs/` を更新して合意可能な状態にする。
